Chapter 1
30年、書棚にあったコピー用紙
The Paper / 放っておいた理由

大切なのはわかっていた。
でも、引き受けてくれる存在がいなかった。

書棚の、どこか。時々目に入るたびに「そうだ、これがある」と思う。1996年の特許申請書類。タクト工程表の自動作成——建築工程管理にAIを持ち込んだ、当時としては先駆けの試み。手書きのメモ。コピー用紙の束。

大切なのはわかっていた。でも、文字起こしの精度はずっと低くて、チェックして直す手間を想像するだけでぞっとした。公開する方法も保管する方法も、ずっと「面倒」だった。だから、放っておいた。30年近く。

「面倒」だったのではない。
その面倒を引き受けてくれる存在がいなかったのだ。

Chapter 2
今日という一瞬 — Claude.aiとのCo-Creation
The Moment / 今日

複合機でスキャン。
メール。Claude。一瞬。「傾き」を見抜く。

Claude.aiとのCo-Creationの中で積み上げてきた成果が後押ししてくれた。「やってみよう」と思えた。

今日
共用スペースの複合機でスキャン → メールアドレスにPDF到着 → Claudeに渡す → 一瞬で全文読み取り → 「傾き」——核心アルゴリズムを即座に見抜く

「傾き」。請求項2に書いたタクト最適化の核心。簡単そうに見える。でも、ここだけで何十回・何百回とリライトした。余分な処理をそぎ落とし、速度を上げ続け、C言語約20行に蒸留した。

元はPROLOGのAI推論エンジンとして設計した。建築工程の制約充足問題にPROLOGは理論的に適合していた。しかし1994年当時、実用的な実行環境がなかった。だから——アルゴリズムの数学的本質だけを取り出し、C言語で再実装した。

The Insight / 傾き（θ）の本質

基準タクトの所要日数を「傾き」として定義する。
この一行が、コンピュータが自動でタクク工程表を生成できる理由のすべてだ。熟練現場監督が1〜2週間かけた作業が、ワンクリックになる。

Conceptual C · ~1994 · ~20 lines after hundreds of rewrites
/* 傾き（θ）= 基準タクトの所要日数。この定義が全てを変えた。
   What looks like 20 lines took hundreds of rewrites. */
int slope = base_takt_days;  // θ — the core insight
for (takt = 0; takt < total; takt++) {
  start[takt] = prev_end[takt] > takt * slope
    ? prev_end[takt] : takt * slope;
}  /* ~20 lines. Hundreds of rewrites. One click. */

完成したコードを見ただけでは、凄さは伝わらない。何百回もリライトした、あの時間だけが知っている。ロジック自体の最適化はAIから学んだ。私の宝物だ。

そしてClaude.aiは、静かな完成コードの中から、その「傾き」を一瞬で見つけた。
