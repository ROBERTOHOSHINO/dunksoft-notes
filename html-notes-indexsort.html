文字列は動かさない。数値だけを動かす。

A sorting technique from the pre-RDB era — that turned out to be the foundation of databases, Git, React, and NoSQL. Some ideas from the frontier never get old.

Roberto Hoshino — 43 years in IT Dunksoft Co., Ltd. — founded 1983 Smart Life Styles™️

Before relational databases were everywhere, before NoSQL was a word, we managed data on flat file systems. Every byte counted. Moving a string in memory was expensive. So we didn't.

"Don't move the data. Move the pointer to the data."

The technique was simple. You load your strings into an array — one line of text per slot. Then you create a second array of integers: just the index numbers, 0, 1, 2, 3... When you need to sort, reorder, or "move" a line, you never touch the strings. You only rearrange the index array. The display reads from the index — so visually, everything looks perfectly sorted. The strings never moved at all.

★ The strings never moved. Zero copy. Zero memory overhead. Just four integers rearranged.
concept.pseudo
// The strings — loaded once, never moved
strings = ["Tokyo", "Osaka", "Nagoya", "Fukuoka"]

// The index — this is all we sort
index   = [0, 1, 2, 3]

// Sort: only rearrange the numbers
index   = [3, 2, 1, 0]   // ← only this changes

// Display: read strings via index
for i in index:
    print(strings[i])   // Fukuoka, Nagoya, Osaka, Tokyo

// strings[] was never touched. Not once.

In the pre-RDB era, this was frontier thinking. We had no query optimizer, no engine doing this for us. We built it by hand because we had to. Memory was precious. Speed was survival.

What surprised me, looking back, is how far this idea traveled.

The same idea — everywhere you look
Database Index (RDB)
A B-tree of pointers to rows. The rows never move. Only the index is sorted. SELECT with ORDER BY is exactly this.
Same idea
Git Object Model
Files are stored as immutable blobs. Commits are just pointers — SHA hashes pointing to trees pointing to blobs. The content never moves.
Same idea
React Virtual DOM
React never touches the real DOM until necessary. It maintains a virtual index of what should be shown, diffs it, then updates only what changed.
Same idea
NoSQL Document ID
In MongoDB, Firestore, DynamoDB — documents sit in place. Queries return IDs (pointers). You fetch by reference, not by scanning content.
Same idea

The frontier era taught us something that classrooms often skip: before you move data, ask whether you need to move it at all. Most of the time, you don't. You just need a better map.

From Prolog to Pascal to Claude — 43 years later, I'm still playing with the same fundamental ideas. They just keep showing up in new clothes.

"The best optimization is the one that never needed to happen."

If you're working with large lists, flat files, or any system where reordering is expensive — try this. Keep your data immutable. Build a lightweight index. Sort the index. The data will appear to move without moving at all.

Works in 1983. Works today.
